# MurderMessageTypes

**Namespace:** Bang.Entities \
**Assembly:** Murder.dll

```csharp
public static class MurderMessageTypes
```

Collection of all ids for fetching messages declared in this project.

`MurderMessageTypes` is compile-time generated by `Bang.Generator` (see `bang\src\Bang.Generator\Templating\Templates.MessageTypes.cs`), which scans the project for every type implementing `IMessage` and emits one `public static int` property per message here, named after the message type (e.g. `InteractMessage` keeps its full name, unlike component properties which strip the `Component` suffix). Each property returns a unique integer index computed as an offset from `MurderComponentsLookup.MurderNextLookupId`, from the same shared index space that [MurderComponentTypes](../../Bang/Entities/MurderComponentTypes.html) draws component indices from. These indices are what [Entity](../../Bang/Entities/Entity.html) uses internally for `HasMessage(int)`, `SendMessage(int, T)` and `RemoveMessage(int)`.

As with `MurderComponentTypes`, you should rarely need these constants directly: [MurderEntityExtensions](../../Bang/Entities/MurderEntityExtensions.html) generates a matching `Has<Message>()`/`Send<Message>(...)`/`Remove<Message>()` trio for every message type that already looks up the right index for you (e.g. `entity.SendInteractMessage()`, `entity.HasCollidedWithMessage()`). Reach for `MurderMessageTypes.SomeMessage` only when you need the raw numeric id itself, such as when working with lower-level, index-based `Entity` APIs.
